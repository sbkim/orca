Planning artifact · Host model follow-up

Tasks, Automations, and Provider Identity Under One Host Model

The project-first migration gives Orca the durable shape Project -> ProjectHostSetup -> Workspace. The remaining product gap is that Tasks, Automations, and provider integrations still route through different ownership rules. This plan makes Host the user-facing concept across Local Mac, SSH machines, Remote Servers, and future Orca cloud VMs.

Host = Local / SSH / Remote Server / Cloud VM Project identity stays above hosts Provider accounts become host-scoped where needed Automations get explicit run location

Short Answer

We should keep the model simple for users: a project can be available on one or more hosts, and every host can have its own filesystem, terminals, provider CLIs, auth, and runtime capabilities. Tasks and Automations should stop relying on ambient global state where the host matters.

The big change is not “make SSH and Remote Server the same internally.” It is “present both as hosts, then route each workflow through the selected host’s capabilities.”

Task Source vs Run Host

A logical project can exist on multiple hosts, but provider data still has to be fetched through one credential context. That source context must be explicit or at least visible. “Project: orca” is not enough information when Local Mac has GitHub account A and openclaw 2 has GitHub account B.

Task source context

Where issues/PRs are fetched from

This answers: which host's GitHub/GitLab/Linear/Jira credentials serve the list? It affects private repo access, rate limits, viewer-specific filters like “assigned to me,” review requests, notifications, and mutation permissions.

Run host

Where the workspace is created

This answers: when the user starts work from a task, which project-host setup creates the workspace? It may default to the task source host, but the user can choose another ready setup when the project exists there.

Recommendation: default the task source to the active workspace host or the last-used host for that project/provider, show the serving account beside the picker, and require an explicit choice when combining hosts would make “me” ambiguous.

What Needs To Change

□
Unify provider request ownership around Host.
GitHub currently has per-repo runtime routing, while SSH/local are desktop-owned. Linear/Jira use active runtime. GitLab uses desktop IPC with SSH-aware repo paths. We need a shared provider request target: { provider, hostId, projectHostSetupId? }.
P0
□
Make Tasks choose an explicit host context.
Applies to Local, SSH, Remote Server, and future Cloud VM hosts. The Tasks page should show which host/account is serving results. Multi-project and multi-host selections should either group by host or require a selected host when provider identity can differ.
P0
□
Store Automations as project + run-host, not repo-id pretending to be project-id.
Existing automations store projectId as a repo id. New automations should target a logical project and either an existing workspace or a projectHostSetupId for fresh workspaces.
P0
□
Add host-scoped provider settings.
Settings should answer “which GitHub/GitLab/Linear/Jira account is active on this host?” without confusing client settings with server/SSH settings.
P0
□
Gate UI by host capabilities and connection state.
Remote Servers can expose runtime RPC capabilities. SSH hosts may support filesystem, terminal, git, and provider CLI execution through Orca’s SSH bridge. Disconnected or incompatible hosts should be visible but disabled where action would fail.
P1
□
Extend CLI/API language to match the product model.
Commands should accept --project plus --host or --project-host-setup for workflows that create workspaces or automations. Keep repo selectors as compatibility aliases.
P1
□
Audit adjacent surfaces that already have host-specific state.
Source-control AI model choices, task resume state, PR/review caches, setup scripts, notification actions, direct launch-from-task flows, and run history should include enough host identity to avoid cross-machine leakage.
P1

Model

Host

One user-facing bucket

Local Mac, SSH target, Remote Server/runtime, and future Cloud VM all appear as hosts. The user chooses where work runs; Orca chooses the transport.

Project setup

Project on a host

A logical project may have setup rows on multiple hosts. Each row owns path, readiness, default worktree location, and setup method.

Provider identity

Account where the request runs

GitHub/GitLab/Linear/Jira identity should be explicit: local desktop account, SSH-host CLI/token, or Remote Server account.

Other Explicitness Gaps

□
Composer “Start from” and linked-work-item picker.
The composer can fetch/link GitHub work items from the selected repo. In the new model it should inherit or show the task source host/account, then choose a separate run host when creating the workspace.
P0
□
Task row actions and issue/PR drawers.
Mutations such as assign, comment, review-request, state change, merge, mark viewed, and create issue need to run under the same explicit provider context that fetched the row, not whichever host is currently focused later.
P0
□
Linear/Jira issue preview pages.
These currently key heavily off active runtime and provider workspace ids. They need a durable source host/account context so opening, editing, commenting, and starting a workspace from an issue stay in the same provider identity.
P0
□
Hosted review / PR helper flows.
Review creation, PR field generation, fix-checks launch, branch-to-PR lookup, and direct launch from review/task need to carry project-host setup plus provider source context. Otherwise the action can resolve against the wrong checkout or account.
P1
□
Settings and integrations entry points.
Provider settings should say whether they are client-global, host-scoped, or project-scoped. Opening settings from Tasks should land on the same host/account context the user was just looking at.
P1
□
Persisted task UI state and caches.
Recent selections, resume state, work item cache keys, PR detail caches, labels, assignees, and partial-failure banners should be keyed by provider source context, not only repo id or active runtime.
P1
□
Automation run history, reuse, and usage attribution.
Runs should store logical project, project-host setup, source provider context, run host, terminal/session ids, and display labels so history remains clear after host disconnects, account changes, or workspace deletion.
P1
□
CLI/API selectors.
CLI commands that create workspaces, create automations, start from tasks, or fetch provider data need explicit --project, --host, and/or --task-source-host options. Existing --repo selectors can remain compatibility aliases.
P1
□
Telemetry and diagnostics.
Events/logs should include non-sensitive host kind and capability status so we can debug routing without recording account names, hostnames, repo paths, or issue text.
P2

Subagent Audit Confirmation

A second codebase pass split across Tasks/provider routing, Automations, Composer/PR launch, and Settings/cache/CLI agreed on the same core gap: Orca needs an explicit TaskSourceContext separate from the ProjectHostSetup used to run work.

Tasks/provider routing

GitHub is partly host-scoped already, but Linear/Jira still route through active runtime, GitLab is mostly repo-path/local-IPC based, and drawer mutations can lose the source account. Report: notes/subagent-tasks-provider-hosts-audit.md.

Automations

Automations still treat projectId as a repo id and do not snapshot hostId, projectHostSetupId, provider source context, or runtime capability/version on runs. Report: notes/subagent-automations-host-model-audit.md.

Composer / PR launch

Start-from flows use one repoId as both source item and run target. Pending worktree creation stores linked numbers but not source host/account context. Report: notes/subagent-composer-pr-launch-hosts-audit.md.

Settings / cache / CLI

Some host foundations are strong, but GitLab selectors are repo-path-only, provider diagnostics are still client-global, agent launch overrides are global, and CLI automation selectors remain repo-first. Report: notes/subagent-settings-cache-cli-hosts-audit.md.

Open Product Decisions

Do Tasks allow multi-host aggregation?

Recommendation: yes for GitHub/GitLab rows when each row can show its serving host, but default to one selected host for Linear/Jira until cross-host grouping is useful.

Should SSH provider auth run on the SSH machine?

Recommendation: eventually yes. In the interim, label desktop-served SSH provider calls explicitly to avoid misleading users with different accounts.

Are Remote Servers “better SSH”?

Recommendation: explain them as richer hosts. SSH gives filesystem/terminal access to a machine you already manage. Remote Server adds Orca runtime services, durable RPC, web/mobile handoff, capability/version negotiation, and richer background behavior.

Do we need host onboarding now?

Recommendation: no. Existing SSH/server settings can remain the entry point. This work should make configured hosts behave correctly once present.
